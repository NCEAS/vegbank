Key:

Actions:
design - database design, not form design. Pretty much everything is affected by this.
implement - implement the database design, that is, take the schema and create the database portion based on the schema. This required reloading all data that are currently in the database.
preload - loading of some portion of the database by VegBank staff, not by users. This is a task, not functionality.
query - the functionality that allows users to view some portion of the database.
userLoad - the functionality of VegBank to allow users to load their data into VegBank.
rectify - during the user loading process, the functionailty VegBank uses to match their data values to extant VegBank records, where possible (i.e. plant concepts, comm concepts, parties, methods).
validate - during the user loading process (could be before the loading process, but the loading process should exist before the validation), the user's data is analyzed and errors identified; plot data verified as good plot data.
alter - functionality for the user to alter extant VegBank records (or add/change a party perspective of extant records).
export - functionality for the user to download VegBank data into various formats for her/his personal use. To program this functionality, the above two steps, validation and rectification, need not be in place. Still, this is the logical place for export on the list.
evaluate - users or PI's or whomever should tell us what is working and not working about our project - both in form and function.
extend - this is for tasks that extend the boundaries of VegBank to serve other organizations, train users, etc. VegBank needs to be stable before investing in this sort of thing heavily.

Aspects:
all - applies across the board to all aspects of the database, primarily meaning plot, plant, and comm modules.
plot - the module of VegBank that deals with plots, including commInterp and taxonInterp type tables.
plant - the plant taxonomy module of VegBank
comm - the community taxonomy module of VegBank
metadata - metadata is usually refering to non-central tables in the plot module of VegBank - methods, parties, citations, revisions, user-defined. These are still critical tables, but are not necessarily loaded at the same time as plots, or through the same means.
userDb - the user database of VegBank users, that is people who have logged onto VegBank.org and have an account.
machine - miscellaneous tasks devoted not primarily to the database itself, but rather the machines that the database runs on.
interface - all tasks dealing with general interface issues, i.e. website.
vegbranch - all tasks that are specifically dealing with VegBranch, the MS-Access version of VegBank. Many VegBranch tasks are listed under plot, plant, or comm tasks if they apply to that portion of VegBranch, as that type of task relates and depends on VegBank tasks.
